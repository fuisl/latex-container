This chapter documents exploratory research on MCP: evaluating it as a way to
standardize model interaction with tools, resources, and application context.
The work informed later design choices but MCP was not adopted as a core
dependency in LangGraph-based development.

%=============================================================================
\section{Why MCP Was Explored}
\label{sec:MCP_Why}
%=============================================================================

Production integration often requires consistent exposure of tools, data, and
context to models. MCP was explored as an open protocol to standardize
discovery/invocation and access to structured resources~\cite{anthropicMCP},
promising interoperability across models and hosts. It matched early
objectives (interfaces, contracts, abstraction value), while its production
maturity remained uncertain.

%=============================================================================
\section{MCP Conceptual Model}
\label{sec:MCP_Conceptual}
%=============================================================================

MCP separates model clients from external systems via an explicit server
boundary. Instead of embedding tool logic into agents, an MCP server advertises
capabilities that models discover and invoke dynamically.

The protocol distinguishes three primary elements:

\begin{itemize}
    \item \textbf{Tools}, which represent executable actions exposed to the
          model through a well-defined interface.
    \item \textbf{Resources}, which provide structured or semi-structured data
          that models can query or retrieve.
    \item \textbf{Context}, which supplies metadata and environmental
          information required to ground model behavior.
\end{itemize}

Clients (LLM runtimes/agent frameworks) need not know application-specific APIs;
they rely on server-described capabilities at runtime, shifting integration to
a protocol-driven interface layer. Interoperability here means reusing the same
tool/resource definitions across model providers and agent implementations
without rewriting integration logic.

\section{Prototype Architecture and Experiments}
\label{sec:MCP_Prototype}
%=============================================================================

To evaluate MCP, a lightweight prototype exposed a limited set of tools and
resources backed by existing application APIs, prioritizing ergonomics,
protocol clarity, and runtime behavior over feature completeness.

The experimental setup consisted of:
\begin{itemize}
    \item An MCP server advertising a small number of application-specific
          tools.
    \item A client-side agent invoking these tools through MCP abstractions.
    \item Manual inspection of tool invocation behavior and response handling.
\end{itemize}

The prototype showed basic interoperability but exposed limitations: tool
invocation depended heavily on model behavior/prompting, and protocol-level
contracts were too weak to guarantee safe usage in complex scenarios.
Security risks were also noted: minimally scoped servers can enable data
exfiltration or cross-tool abuse~\cite{croce2025trojans,song2025attack},
reinforcing the need for governance and validation in production.

\section{Outcomes and Lessons}
\label{sec:MCP_Outcomes}
%=============================================================================

MCP exploration informed later design: the abstraction aids interoperability
but introduces trade-offs among flexibility, safety, and determinism.

\begin{table}[htbp]
    \centering
    \caption{Findings from MCP exploration: assumptions, experiments, and lessons learned.}
    \label{tab:MCP_Findings}
    \begin{tabular}{p{3cm}p{3.2cm}p{3.2cm}p{4.2cm}}
        \toprule
        \textbf{Assumption}                                            &
        \textbf{Experiment Conducted}                                  &
        \textbf{Observed Result}                                       &
        \textbf{Lesson Learned}                                                                             \\
        \midrule
        Tool interfaces can be fully standardized                      &
        Implemented MCP-based tool exposure for agents                 &
        Tool invocation behavior varied significantly across models    &
        Protocol definitions require stronger behavioral constraints, not just schema-level standardization \\
        \midrule
        Interoperability reduces integration effort                    &
        Reused MCP-exposed tools across multiple agents                &
        Integration became simpler, but debugging complexity increased &
        Abstractions shift complexity rather than eliminating it                                            \\
        \midrule
        MCP is production-ready                                        &
        Performed a security-oriented review of the prototype          &
        Multiple potential attack vectors were identified              &
        Governance and security controls are mandatory for safe production adoption                         \\
        \bottomrule
    \end{tabular}
\end{table}

Key outcome: protocol-level abstractions reduce coupling and improve reuse, but
do not replace explicit control over execution flow, validation, and safety.
This motivated a shift toward structured orchestration with explicit states and
paths. MCP served as a research probe, clarifying limits and contrasting with
later stateful, workflow-oriented agent systems.
