This chapter documents exploratory research on the Model Context Protocol
(MCP): evaluating it as a standardization layer for model-to-tool interaction.
MCP informed later design choices but was not adopted as a core dependency in
production systems.

%=============================================================================
\section{Why MCP Was Explored}
\label{sec:MCP_Why}
%=============================================================================

Early agent prototypes often couple tools tightly to frameworks. MCP
~\cite{anthropicMCP} offers an open protocol to standardize tool discovery,
invocation, and resource access across models and platforms, promising
interoperability without rewriting integration logic. The exploration assessed
whether protocol-level abstraction simplifies agent integration and whether MCP
was production-ready.

%=============================================================================
\section{MCP Conceptual Model}
\label{sec:MCP_Conceptual}
%=============================================================================

MCP separates model clients from external systems via an explicit server that
advertises capabilities dynamically.

The protocol distinguishes three primary elements:

\begin{itemize}
    \item \textbf{Tools}, which represent executable actions exposed to the
          model through a well-defined interface.
    \item \textbf{Resources}, which provide structured or semi-structured data
          that models can query or retrieve.
    \item \textbf{Context}, which supplies metadata and environmental
          information required to ground model behavior.
\end{itemize}

Clients (LLM runtimes/agent frameworks) rely on server-described capabilities,
shifting integration to a protocol-driven layer. Interoperability means reusing
tool/resource definitions across providers without rewriting logic.

\section{Prototype Architecture and Experiments}
\label{sec:MCP_Prototype}
%=============================================================================

A lightweight prototype exposed application-specific tools via MCP server,
invoked by a client-side agent. Basic interoperability worked, but limitations
emerged: tool invocation heavily depended on model behavior/prompting; weak
protocol-level contracts couldn't guarantee safe usage; and security reviews
identified data exfiltration/cross-tool abuse
risks~\cite{croce2025trojans,song2025attack}. Governance and validation are
mandatory for production use.

\section{Outcomes and Lessons}
\label{sec:MCP_Outcomes}
%=============================================================================

MCP exploration clarified interoperability trade-offs: abstraction reduces
coupling but introduces safety and determinism challenges.

\begin{table}[htbp]
    \centering
    \small
    \caption{MCP exploration findings.}
    \label{tab:MCP_Findings}
    \begin{tabular}{p{4cm}p{4.5cm}p{5cm}}
        \toprule
        \textbf{Assumption}                 & \textbf{Observation}                     & \textbf{Lesson}                              \\
        \midrule
        Tool interfaces can be standardized & Invocation behavior varied across models & Need behavioral constraints, not just schema \\
        Interoperability reduces effort     & Simpler integration, harder debugging    & Abstractions shift complexity                \\
        MCP is production-ready             & Security review found attack vectors     & Governance and security controls mandatory   \\
        \bottomrule
    \end{tabular}
\end{table}

Key learning points:
\begin{itemize}
    \item Protocol abstractions aid reuse but require stronger behavioral constraints beyond schema-level definitions.
    \item Interoperability shifts complexity (simpler integration, harder debugging) rather than eliminating it.
    \item MCP requires governance/security controls for production; minimally scoped servers pose exfiltration risks.
    \item Explicit control over execution flow, validation, and safety cannot be replaced by protocol-level abstraction alone.
\end{itemize}

These findings motivated the shift toward structured orchestration (stateful,
workflow-oriented systems) explored in Track B. MCP served as a useful research
probe, contrasting with later agent platform approaches.
