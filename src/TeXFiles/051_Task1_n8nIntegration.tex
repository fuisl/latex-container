\chapter{ERP Integration via n8n}
\label{ch:Task1_n8n}
%=============================================================================
% Page budget: 5 pages (Pages 13--17)
%=============================================================================

This chapter describes an \gls{erp} integration built with \gls{n8n} to add
AI-assisted steps while keeping the workflow control plane deterministic,
observable, and safe for production~\cite{n8nDocs}. The focus is on
principles: explicit control flow, bounded AI calls (where the model selects
tools), and recoverable failure paths.

%=============================================================================
\section{Problem Statement and Requirements}
\label{sec:Task1_Problem}
%=============================================================================

ERP integrations are sensitive to errors and partial failures. The goal was to
insert AI-assisted steps without sacrificing stability: react to ERP events,
invoke AI in a bounded node (the model may choose tools), and apply actions
back with full traceability. Key constraints: deterministic workflow control,
failure isolation, auditability, and future extensibility. Agent-style
autonomy and cross-run learning were explicitly out of scope.

%=============================================================================
\section{Workflow Design}
\label{sec:Task1_WorkflowDesign}
%=============================================================================

Workflow design (n8n): event trigger → validation → bounded AI call →
deterministic branching → ERP update. The AI node can choose which tool to
invoke, but is constrained by structured output contracts and validation to
contain non-determinism. Failure handling is first-class: retries with backoff
for transients, abort/compensate for hard errors, and safe replay to prevent
duplicates~\cite{googleSRE}.

\section{Operational Considerations}
\label{sec:Task1_Operations}
%=============================================================================

Operational principles: externalized secrets for environment portability;
idempotent ERP interactions for safe retries/replays; and rate limits/throttles
at boundaries to protect both ERP and AI services~\cite{googleSRE,microsoftWellArchitected}.

%=============================================================================
\section{Results and Learning Points}
\label{sec:Task1_Results}
%=============================================================================

Results: AI steps can be added without losing determinism or safety when the
control plane stays explicit. Limitations: complex AI logic strains workflow
maintainability; adaptive behavior becomes cumbersome with branching.

Key learning points:
\begin{itemize}
    \item Keep AI calls bounded and typed; let workflows own control and recovery.
    \item Design for retries/replay with idempotent side effects and compensation paths.
    \item Use workflow tools for safety and auditability; move adaptive logic to agents when complexity grows.
\end{itemize}
