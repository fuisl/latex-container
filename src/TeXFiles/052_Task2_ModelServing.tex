%=============================================================================
\chapter{Local Model Serving and Evaluation}
\label{ch:LocalModelServing}
%=============================================================================
% Page budget: 4--5 pages
%=============================================================================

This chapter describes the exploration of running large language models (LLMs)
locally to understand performance limits and trade-offs. The goal was not to build
a production system, but rather to understand feasibility, speed, and hardware
constraints. The exploration progressed from simple tools like Ollama to vLLM for
more realistic testing, informing decisions about using managed services.

%=============================================================================
\section{Objectives and Scope}
\label{sec:LocalServing_Objectives}
%=============================================================================

Local model serving was investigated to answer three questions:

\begin{itemize}
      \item Can LLM inference run cheaply without cloud APIs?
      \item How fast is local inference, and how much GPU memory does it use?
      \item Can one affordable GPU handle interactive workflows with multiple users?
\end{itemize}

The scope was limited to single machines. High availability, autoscaling, and
multi-region support were not addressed.

%=============================================================================
\section{Model Selection}
\label{sec:LocalServing_ModelSelection}
%=============================================================================

Two categories of models were used:

\begin{itemize}
      \item \texttt{gpt-oss:20b} and \texttt{qwen2.5-7b-instruct} for generation: a
            reasoning and a non-reasoning model, both transparent and self-hosted~\cite{openai2025gptoss,qwen2.5}.
      \item \texttt{nomic-embed-text} and \texttt{bge-m3} for RAG: embedding models that convert
            documentation into vectors, keeping the entire pipeline
            local~\cite{nomic2025embed,bgem3}.
\end{itemize}

%=============================================================================
\section{Serving Architecture Evolution}
\label{sec:LocalServing_Architecture}
%=============================================================================

\subsection{Initial Prototyping with Ollama}

\texttt{Ollama}, a simple runtime for running open-source LLMs on local
hardware, was initially used~\cite{ollama}. It provided an easy HTTP API and
quick setup, making it suitable for testing prompts and agent logic.

However, Ollama's performance degraded with multiple concurrent users and
provided limited visibility into GPU memory and compute usage.

\subsection{Transition to vLLM}

The exploration transitioned to \texttt{vLLM}, a high-throughput inference
engine, to evaluate realistic performance~\cite{kwon2023efficient}. vLLM
incorporates two key techniques:

\begin{itemize}
      \item \textit{PagedAttention}: Reduces GPU memory waste from KV caches.
      \item Continuous batching: Runs multiple requests in parallel to use GPU
            fully.
\end{itemize}

As shown in Figure, the vLLM setup included:

\begin{itemize}
      \item containerized vLLM instances deployed via Docker and Docker Compose,
      \item multiple model endpoints exposed on separate ports,
      \item an \texttt{Nginx} reverse proxy for per-model request routing,
      \item secure external access provided through \texttt{Tailscale Funnel}.
\end{itemize}

\begin{figure}[htbp]

      \includegraphics[width=\textwidth]{Figures/model-serving.png}
      \caption{vLLM serving architecture diagram illustrating the main components and their interactions.}
      \label{fig:vllm-serving}
\end{figure}

This setup allowed controlled tests of concurrent users, batching, and multiple
models on one GPU, while staying operationally simple.

%=============================================================================
\section{Hardware Experiments and Constraints}
\label{sec:LocalServing_Hardware}
%=============================================================================

Observations:
\begin{itemize}
      \item RTX 3060 laptop: Not enough memory. Both tools ran slowly and crashed.
      \item RTX 4090 (24~GB): vLLM worked well. GPU memory for KV caches is the main limit on context length and how many users can run at once~\cite{pope2022efficiently}.
      \item Multiple models on one GPU need careful tuning of batching and cache limits to stay responsive.
\end{itemize}

%=============================================================================
\section{Evaluation and Outcomes}
\label{sec:LocalServing_Evaluation}
%=============================================================================

Evaluation focused on speed, stability, and iteration velocity. Local serving
proved suitable for early testing, but vLLM alone is not production-ready. It
lacks elasticity, built-in failover, and latency guarantees required by
production workloads, motivating a transition to managed services.

vLLM is much faster than Ollama: roughly 3400 tokens per second for input and
300 for output. This matches similar benchmarks from RedHat's team (see figures
below).

\section{Key Learning Points}
\label{sec:LocalServing_Learnings}
%=============================================================================

Several lessons emerged from this exploration:

\begin{itemize}
      \item Local serving lowers the barrier to experimentation but does not
            remove fundamental performance and scaling constraints.
      \item vLLM provides substantial improvement over lightweight runtimes by
            enabling efficient batching and KV-cache management, making realistic
            concurrency experiments possible.
      \item Serving architecture choices strongly influence downstream system
            design, particularly for interactive and human-in-the-loop agents.
\end{itemize}

Local model serving proved valuable as an exploratory and evaluative tool,
shaping informed decisions about production architecture rather than serving as
a production endpoint itself.
